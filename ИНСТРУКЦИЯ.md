# ШПАРГАЛКА — Что показывать на экзамене

---

## ЗАПУСК

```bash
python3 server.py   # терминал 1
python3 client.py   # терминал 2
```

Логин: `admin` / `Admin@123`

---

## ЧТО СПРОСИТ ПРЕПОДАВАТЕЛЬ → ЧТО ПОКАЗАТЬ

---

### «Покажи аутентификацию»
> **п. 3.2.1** — смена пароля при первом входе
> **п. 3.2.1** — пароль скрыт при вводе
> **п. 3.2.1** — блокировка после 3 попыток на 15 минут

1. Открыто окно входа — **пароль скрыт звёздочками**
2. Введи неверный пароль 3 раза → сообщение **«Аккаунт заблокирован на 15 мин»**
3. Войди как `admin` / `Admin@123`
4. Войди как `user1` / `User@12` → сразу откроется **принудительная смена пароля**

---

### «Покажи требования к паролю»
> **п. 3.2.1** — длина и сложность пароля для пользователей и администраторов
> **п. 3.2.1** — пароль не совпадает с именем пользователя

При смене пароля введи слабый пароль (например `123`) → система покажет список ошибок:
- минимум **6 символов** (пользователь) / **7 символов** (администратор)
- нужна заглавная буква, строчная, цифра, спецсимвол (`~@%&*$^!#`)
- пароль не должен содержать имя пользователя

Кнопка «Сменить пароль» — в **верхней синей панели** главного окна.

---

### «Покажи расчёт»
> **п. 1.1.2.1** — расчёт при заданных: нагрузка, блоки, температура, влажность, ветер
> **п. 1.1.2.2** — определение оптимального режима работы блока
> **п. 1.1.2.3** — визуализация графиков зависимости от температуры
> **п. 2** — математическое ядро из Приложения 1

Вкладка **«Расчёт»**:
1. Заполни поля (нагрузка, блоки, температура, влажность, ветер)
2. Нажми **«Рассчитать»**
3. Справа появятся результаты: КПД блока, КПД брутто, собственные нужды, КПД нетто, удельный расход топлива
4. Вкладка **«График»** — зависимость КПД от температуры −20…+40°C
5. Вкладка **«Оптимизация числа блоков»** — таблица + столбчатый график, оптимальный вариант выделен зелёным

---

### «Как учитываются внешние факторы?»
> **п. 1.1.2.1** — поправки на температуру, влажность, направление и скорость ветра
> **п. 2** — особое внимание поправкам внешних факторов

Объяснить: в полях ввода есть **температура, влажность, скорость и направление ветра** — каждый параметр влияет на поправочный коэффициент, который корректирует КПД. Поменяй температуру с +15 на +35 и пересчитай — КПД упадёт, расход топлива вырастет.

---

### «Покажи управление пользователями»
> **п. 3.2.1** — администратор устанавливает пароли пользователей
> **п. 3.3.1** — действия администратора по изменению прав доступа регистрируются

Вкладка **«Администрирование» → «Пользователи»** (только у admin):
- список всех пользователей, их роль, статус, попытки входа
- кнопка **«Создать пользователя»** → задать логин, пароль, роль
- кнопка **«Вкл/Выкл аккаунт»** → деактивировать/активировать
- кнопка **«Разблокировать»** → снять блокировку после 3 попыток
- кнопка **«Сбросить пароль»** → пользователь обязан сменить при следующем входе

---

### «Покажи журнал событий / аудит»
> **п. 3.3.1** — какие события регистрируются
> **п. 3.3.2** — что содержит каждая запись

Вкладка **«Администрирование» → «Журнал событий»**:

Показать таблицу — в каждой строке видны все поля из **п. 3.3.2**:

| Колонка | Поле из п. 3.3.2 |
|---|---|
| `#` | порядковый номер события |
| `Время` | дата и время события |
| `Тип события` | наименование и тип события |
| `Компонент` | объект доступа (auth, calc_engine) |
| `Пользователь` | субъект доступа |
| `IP` | идентификация стороны взаимодействия |
| `Служебные заголовки` | служебные заголовки (метод, путь, User-Agent) |
| `Детали` | идентификатор события, дополнительный контекст |

> Нажать «Обновить журнал» — новые события появятся сразу.

---

### «Что журнал НЕ содержит?»
> **п. 3.3.3** — журнал не должен содержать пароли и персональные данные

Ответить: пароли (ни в открытом, ни в хэшированном виде), персональные данные (ФИО, дата рождения, адрес), токены аутентификации. Показать столбец «Детали» — там только роль, номер попытки, нагрузка.

---

### «Покажи настройку глубины журнала»
> **п. 3.3.4** — глубина хранения и детализация должны быть настраиваемыми

Вкладка **«Администрирование» → «Настройки журнала»**:
- **Макс. размер файла** — порог ротации (объём)
- **Кол-во архивных копий** — глубина хранения по числу файлов
- **Срок хранения** — глубина хранения по времени (в днях)
- **Уровень детализации** — ALL / DEBUG / INFO / WARNING / ERROR
- Нажать **«Применить»** — изменения вступают в силу немедленно

Показать переключение уровня: INFO → ALL → INFO.

---

### «Что значат уровни детализации?»
> **п. 3.3.4** — детализация должна быть настраиваемой

| Уровень | Что попадает в журнал |
|---|---|
| ALL | Абсолютно все события всех уровней |
| DEBUG | Всё + технические запросы |
| INFO | Входы, расчёты, смена паролей (стандарт) |
| WARNING | Только неверные пароли |
| ERROR | Только блокировки аккаунтов |

---

### «Покажи ротацию журнала»
> **п. 3.3.6** — возможность работы в режиме ротации при ограничении места хранения

Объяснить: при достижении максимального размера файл автоматически архивируется, хранится заданное число копий, старые удаляются. Настраивается в «Настройках журнала» — поля «Макс. размер файла» и «Кол-во архивных копий».

---

### «Покажи удалённую отправку логов»
> **п. 3.3.5** — возможность отправки журналов на удалённый сервер

В «Настройках журнала» поле **«URL удалённого сервера логов»** — вписать адрес сервера. Пустое поле = отключено.

---

### «Покажи защиту от перебора паролей / DDoS»
> **п. 4.1.3** — ограничение частоты запросов для предотвращения DDoS и перебора паролей

Объяснить: на эндпоинт входа установлен **rate limit 20 запросов/минуту** (slowapi). При превышении сервер возвращает ошибку 429. Плюс прикладная блокировка аккаунта после 3 попыток (**п. 3.2.1**).

---

### «Как регистрируются API-запросы?»
> **п. 4.1.3** — регистрация всех событий, связанных с запросами к API, включая аутентификацию

Показать в журнале событий колонку **«Служебные заголовки»** — там видны метод, путь, User-Agent каждого запроса к API. Каждое обращение через интерфейс фиксируется.

---

### «Покажи историю расчётов»
> **п. 3.3.1** — действия пользователей, связанные с изменением данных, регистрируются

Вкладка **«История»** — таблица всех расчётов текущего пользователя: дата, нагрузка, блоки, температура, КПД нетто, расход топлива.

---

### «Покажи выход из системы»
> **п. 3.3.1** — выход пользователей регистрируется в журнале

Кнопка **«Выйти»** в верхней синей панели → токен отзывается на сервере (в журнале появляется запись `LOGOUT`) → возврат к окну входа.
