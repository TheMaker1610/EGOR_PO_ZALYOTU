# Программный комплекс «Система оптимизации ТЭС»
## Техническое описание и инструкция по эксплуатации

---

## 1. Назначение программного комплекса

Программный комплекс предназначен для оперативного расчёта режимов работы тепловых электростанций (ТЭС) с учётом внешних факторов среды. Система позволяет:

- рассчитывать основные энергетические показатели энергоблоков;
- определять оптимальное количество работающих блоков при заданной нагрузке;
- визуализировать зависимость теплоотводящей способности от температуры наружного воздуха;
- вести защищённый журнал регистрации всех событий;
- управлять пользователями и их правами доступа.

---

## 2. Архитектура программного комплекса

Система состоит из следующих взаимосвязанных компонентов:

| Компонент | Реализация | Назначение |
|---|---|---|
| Клиентская часть | PyQt5 (`client.py`, `gui/`) | Графический интерфейс пользователя |
| Серверная часть | FastAPI + Uvicorn (`ё/`, `server.py`) | REST API, оркестрация запросов |
| Математическое ядро | Python (`core/math_engine.py`) | Расчёт показателей ТЭС |
| Подсистема аутентификации | JWT + bcrypt (`auth/`) | Авторизация и управление сессиями |
| Подсистема хранения данных | SQLite + SQLAlchemy (`database/`) | Хранение пользователей, сессий, расчётов |
| Подсистема аудита и логирования | Python logging (`logging_subsystem/`) | Журнал регистрации событий |
| Оркестратор задач | FastAPI роутеры (`ё/routers/`) | Маршрутизация и rate limiting |

### Стек технологий

- **Язык:** Python 3.10+
- **Сервер:** FastAPI, Uvicorn
- **GUI:** PyQt5, Matplotlib
- **БД:** SQLite (через SQLAlchemy ORM)
- **Аутентификация:** JWT (python-jose), bcrypt (passlib)
- **Ограничение запросов:** slowapi

### Установка зависимостей

```bash
pip install -r requirements.txt
```

**Способ 1 — одной командой:**

## 4. Руководство пользователя

### 4.1. Вход в систему

При запуске клиента открывается окно входа. Введите логин и пароль. Символы пароля скрыты (`***`).

- При **неверном пароле** система показывает количество оставшихся попыток.
- После **3 неверных попыток** аккаунт блокируется на 15 минут автоматически.
- Заблокированный аккаунт разблокируется автоматически по истечении времени или вручную администратором.

### 4.2. Смена пароля

- **Принудительная** — при первом входе (если администратор установил флаг смены).
- **Добровольная** — кнопка «Сменить пароль» в верхней панели главного окна.

### 4.3. Выход из системы

Кнопка «Выйти» в верхней панели главного окна. После выхода открывается окно входа для повторной аутентификации.

## 5. Расчёт показателей ТЭС

### 5.1. Входные параметры

| Параметр | Единица | Допустимый диапазон |
|---|---|---|
| Электрическая нагрузка ТЭС | МВт | 1 — 10 000 |
| Количество работающих блоков | шт. | 1 — 20 |
| Температура наружного воздуха | °C | −50 — +60 |
| Влажность наружного воздуха | % | 0 — 100 |
| Скорость ветра | м/с | 0 — 50 |
| Направление ветра | градусы | 0 — 360 |
| Номинальная мощность блока | МВт | 1 — 2 000 |
| Номинальный КПД | доли ед. | 0.01 — 1.0 |
| Коэффициент собственных нужд | доли ед. | 0 — 0.3 |

### 5.2. Выходные данные

| Показатель | Единица |
|---|---|
| Нагрузка на энергоблок | МВт |
| КПД блока | % |
| КПД ТЭС брутто | % |
| Собственные нужды | МВт / % |
| КПД ТЭС нетто | % |
| Удельный расход топлива | г у.т./кВт·ч |

### 5.3. Математическое ядро (Приложение 1)

Расчёт базируется на следующих поправочных коэффициентах:

**Поправка на температуру воздуха:**
- При t ≤ 15°C: `k_t = 1 + 0.003 × (15 − t)`
- При t > 15°C: `k_t = 1 − 0.002 × (t − 15)`

**Поправка на влажность:**
- При φ ≤ 60%: `k_φ = 1.0`
- При φ > 60%: `k_φ = 1 − 0.0005 × (φ − 60)`

**Поправка на скорость ветра:**
- При v ≤ 8 м/с: `k_v = 1 + 0.002 × v`
- При v > 8 м/с: `k_v = 1 + 0.002 × 8`

**Поправка на направление ветра:**
- 0–45° (со стороны трубы): `k_dir = 0.99`
- 180–225° (встречный): `k_dir = 1.01`
- Прочие направления: `k_dir = 1.00`

**Суммарный коэффициент:** `k = k_t × k_φ × k_v × k_dir`

**КПД блока:** `η_block = η_nom × k × (1 − β × (1 − P/P_nom)²)`

**Собственные нужды** корректируются по температуре:
- При t > 25°C: `α = α_0 + 0.005 × (t − 25)`
- При t < 0°C: `α = α_0 + 0.003 × |t|`

**КПД нетто:** `η_netto = η_brutto × (1 − α)`

**Удельный расход топлива:** `b = 123 / η_netto` (г у.т./кВт·ч)

### 5.4. Оптимизация числа блоков

Вкладка «Оптимизация числа блоков» автоматически рассчитывает эффективность при разном числе работающих блоков (от 1 до 6) и выделяет зелёным оптимальный вариант с максимальным КПД нетто.

### 5.5. График зависимости от температуры

Вкладка «График» строит зависимость КПД нетто и удельного расхода топлива от температуры наружного воздуха в диапазоне от −20°C до +40°C с шагом 5°C.

## 6. Требования к аутентификации (п. 3.2 ТЗ)

### 6.1. Скрытие пароля при вводе
При вводе пароля в любом окне системы все символы заменяются на `*`. Реализовано через `QLineEdit.EchoMode.Password`.

### 6.2. Требования к паролю

| Роль | Мин. длина | Верхний регистр | Нижний регистр | Цифра | Спецсимвол |
|---|---|---|---|---|---|
| Пользователь | 6 символов | ✅ | ✅ | ✅ | ✅ |
| Администратор | 7 символов | ✅ | ✅ | ✅ | ✅ |

Допустимые спецсимволы: `~ @ % & * $ ^ ! #`

Пароль не должен содержать имя пользователя (в любом регистре).

### 6.3. Блокировка аккаунта

- Максимальное количество неуспешных попыток: **3**
- При превышении — аккаунт блокируется на **15 минут**
- Автоматическая разблокировка по истечении времени блокировки
- Ручная разблокировка администратором: «Администрирование» → «Пользователи» → «Разблокировать»

### 6.4. Смена пароля при первом входе
Пользователь, созданный администратором, при первом входе принудительно перенаправляется на экран смены пароля. До смены пароля доступ к функциям системы закрыт.

## 7. Журнал регистрации событий (п. 3.3 ТЗ)

### 7.1. Регистрируемые события (п. 3.3.1)

| Событие | Тип записи |
|---|---|
| Успешный вход | `LOGIN_SUCCESS` |
| Неудачная попытка входа | `LOGIN_FAILED` |
| Блокировка аккаунта | `LOGIN_LOCKED` |
| Попытка входа в заблокированный аккаунт | `LOGIN_BLOCKED` |
| Выход из системы | `LOGOUT` |
| Успешная смена пароля | `CHANGE_PASSWORD_SUCCESS` |
| Неудачная попытка смены пароля | `CHANGE_PASSWORD_FAILED` |
| Выполнение расчёта | `CALCULATION` |
| Создание пользователя | `USER_CREATED` |
| Изменение данных пользователя | `USER_UPDATED` |
| Сброс пароля администратором | `PASSWORD_RESET` |
| Разблокировка пользователя | `USER_UNLOCKED` |

### 7.2. Структура записи журнала (п. 3.3.2)

Каждая запись содержит все обязательные поля:

| Поле | Описание | Пример |
|---|---|---|
| Дата и время | Момент события | `2026-05-01 12:45:53` |
| Порядковый номер | Автоинкрементный ID в БД | `1` |
| Идентификатор события | UUID v4 | `b41862a6-16e2-...` |
| Тип события | Категория | `LOGIN_SUCCESS` |
| Наименование события | Совпадает с типом | `LOGIN_SUCCESS` |
| Компонент (объект доступа) | Подсистема | `auth`, `calc_engine` |
| Учётная запись | Пользователь | `admin` |
| Служебные заголовки | HTTP метод, путь, User-Agent | `method=POST; path=/auth/login; host=...; user-agent=...` |
| Детали | Дополнительный контекст | `role=admin`, `attempt 2` |

**Пример записи в лог-файле:**
```
2026-05-01 12:45:53 | INFO | tes_app | EVENT | id=b41862a6-... | type=LOGIN_SUCCESS | component=auth | user=admin | ip=127.0.0.1 | headers=[method=POST; path=/auth/login; host=127.0.0.1:8000; user-agent=TES-Client/1.0; content-type=application/json] | role=admin
```

### 7.3. Защита содержимого журнала (п. 3.3.3)

Журнал **не содержит:**
- паролей в открытом или хэшированном виде;
- персональных данных (ФИО, дата рождения, адрес и т.д.);
- токенов аутентификации.

### 7.4. Настраиваемая глубина хранения и детализация (п. 3.3.4)

Настройки доступны в интерфейсе: **«Администрирование» → «Настройки журнала»**

| Параметр | Описание | Значение по умолчанию |
|---|---|---|
| Макс. размер файла | Порог ротации журнала | 5 МБ |
| Количество архивных копий | Глубина хранения по объёму | 5 файлов |
| Срок хранения записей | Глубина хранения по времени | 30 дней |
| Уровень детализации | Фильтр событий по важности | INFO |
| URL удалённого сервера | Адрес для отправки логов | (пусто) |

**Уровни детализации:**

| Уровень | Что фиксируется |
|---|---|
| **ALL** | Абсолютно все события всех уровней |
| **DEBUG** | Технические запросы + все события аудита |
| **INFO** | Входы, выходы, расчёты, смена паролей, создание пользователей |
| **WARNING** | Неверные пароли, неудачные попытки смены пароля |
| **ERROR** | Только блокировки аккаунтов |

Изменения применяются немедленно без перезапуска сервера.

### 7.5. Отправка журналов на удалённый сервер (п. 3.3.5)

Настраивается через поле «URL удалённого сервера логов» в разделе «Настройки журнала». При пустом значении — удалённая отправка отключена.

### 7.6. Ротация журналов (п. 3.3.6)

Реализована через `RotatingFileHandler`:
- при достижении максимального размера файл автоматически переименовывается в архив;
- хранится заданное количество архивных копий;
- старые копии удаляются автоматически.

## 8. Требования к интеграционным потокам (п. 4.1 ТЗ)

### 8.1. Безопасность передачи данных (п. 4.1.1–4.1.2)

- Все запросы между клиентом и сервером передаются по HTTP с JWT-токеном в заголовке `Authorization: Bearer <token>`.
- Токен подписан алгоритмом **HS256**, срок действия — 60 минут.
- Каждая сессия имеет уникальный идентификатор (`jti`), хранится в БД и может быть отозвана при выходе.
- Перехват токена исключается отзывом сессии при выходе (`revoked = True`).

### 8.2. Ограничение частоты запросов (п. 4.1.3)

| Эндпоинт | Лимит |
|---|---|
| `/auth/login` | 20 запросов/минуту (защита от перебора паролей) |
| `/calc/run` | 60 запросов/минуту |
| Остальные | без ограничений на уровне приложения |

### 8.3. Регистрация API-запросов (п. 4.1.3)

Все обращения к API, включая информацию об аутентификации и авторизации, регистрируются в журнале аудита с указанием служебных заголовков (метод, путь, User-Agent).
